### 0-1. スコープ

機能を4分解したうち、PoCは **(a)〜(c)** を対象とする。

| # | 機能 | PoC | 備考 |
|---|---|---|---|
| (a) | ナレッジ検索による自己解決UI（引用付きRAG回答） | ✅ **対象・本命** | 質問→F3自社ナレッジを検索→「どの資料のどの部分か」を提示して回答 |
| (b) | 問い合わせの起票・管理 | ✅ 対象（最小限） | 自己解決できない場合にフォームから起票。ステータスは最小限 |
| (c) | AIによる回答ドラフト生成（NuRO向け） | ✅ 対象 | 起票された問い合わせに対しAIがドラフト提示、NuRO担当者が参考に回答 |
| (d) | 回答のナレッジ自動登録（ナレッジFMT＝**F2**への書き戻し） | ❌ **範囲外** | 本番構想。NuRO設立間もなくナレッジ管理体制が整備中のため |

**その他の範囲外**：添付資料（フォームのファイルアップロード）／過去問い合わせ（メール・電話ログ）の
シードデータ投入／メール等の通知連携。

### 0-2. ユーザー

- **PoCユーザー＝電力のみ**（NuRO担当者は (c) の回答オペレーションで登場するが、UIの主対象は電力ユーザー）。
- 認証・ユーザー識別の方式は未確定（§9）。PoCでは既存プラットフォームと同等の簡易運用でよい。

### 0-3. 検索対象（リトリーバル範囲）

- **PoC＝F3自社ナレッジのみ**（事前レビュー Tool2a と同一のデータ範囲。
  問い合わせ元の電力会社＝自社分に限定。他社F3は検索対象外）。
- **本番構想**＝廃炉全般・あらゆる社内資料への横断ナレッジ検索（RAG）。
  PoCの設計は本番拡張（データストア追加・資料種別の増加）を阻害しない形をとる（§5）。
- 問い合わせ内容の想定は**廃炉全般**（転記に限らない）。ただしPoC評価はF3に答えがある領域で行う。

### 0-4. コア要件（本PoCの検証命題）

> **「ナレッジとして存在するものはハルシネーションせずに引用付きで回答し、
> 存在しないものは『ない』と判定してNuROへの問い合わせに流せる」** システムであること。

分解すると：

1. **引用の構造的保証**：回答には「どの資料（ファイル・シート・レコード）のどの部分か」の根拠を必ず付す。
   引用はプロンプト指示による自由記述ではなく、**検索結果に紐づく構造化データ**として保証する（§4-2）。
2. **棄却（abstention）の構造的保証**：ナレッジに答えがない質問には回答を生成せず、
   「ナレッジに存在しない」と明示して**起票導線（(b)）に接続**する。
   棄却判断はモデルの自制に頼らず、**パイプラインの独立ステップ**として実装する（§4-3）。

### 0-5. 横断原則（事前レビューと共通）

- チェック・検索の拠り所を特定の費目・資料構造・検証ケースに**ハードコード/過剰適合しない**。
- 既存の `run_review` / `knowledge_loader` の **I/F は不変**（本機能からの利用は読み取りのみ。
  変更が必要な場合もデフォルト付きオプション引数の追加まで）。
- ドキュメントは本ディレクトリ（`docs/inquiry/`）に集約し、**勝手に増やさない**。
- 一時生成物（評価レポート等）はコミットしない。ダミー他社名は「北の海電力」。

## 1. 機能概要・背景

提案資料「問合せナレッジ対応自動化」（7102・ステップ4シナリオ）に基づく。

現状（ステップ3まで）の課題：

| フロー | 現状 | 課題 |
|---|---|---|
| 自己解決 | ファイル検索で情報を探す | 検索対象が広く、必要情報に辿り着くまで時間がかかる |
| 問合せ | メール/電話 | 問い合わせの段階が不明・プロセス全体が不透明 |
| 問合せ受付 | メール/電話で受付 | 情報がチャネルごとに散在 |
| 問合せ解決 | NuRO内部で検討・ファイル検索 | 調査と回答作成に時間と工数がかかる |
| ナレッジ登録 | ナレッジFMTへ手動転記 | 追加作業が発生（→ (d)・PoC範囲外） |

解決の方向性：検索対象を**ナレッジに絞り込んだRAG**で自己解決を促進し、
未解決分は**システムで起票**して一元管理、NuROの回答は**AIドラフト**で工数削減。

期待効果：電力＝必要情報への到達時間短縮・自己解決率向上／NuRO＝重複問い合わせの抑制・対応工数削減。

## 2. ユーザーフロー（PoC版）

提案資料の処理フロー（1〜17）のうち、PoC範囲 (a)(b)(c) を抜粋・簡略化：

```
[電力] 1. 質問を入力
   → [AI] 2. F3自社ナレッジを検索し、十分性を判定（§4-3）
        ├─ 十分 → 3. 引用付き回答＋根拠資料を即時表示
        │          [電力] 4. 自己解決を試す
        │             ├─ 解決 → 5. 終了
        │             └─ 未解決 ↓
        └─ 不十分 → 「ナレッジに存在しません」と明示 ↓
   [電力] 6. 問い合わせを登録（質問内容をプリフィル）
   → [システム] 7. 問い合わせを保存（Firestore）
   → [NuRO] 8-9. 問い合わせ内容＋AIドラフト（§3-3）を確認し、回答を作成
   → [システム] 10. 回答を保存
   → [電力] 11-12. 回答を確認 → 解決（未解決なら再起票）
（13〜17 ナレッジ化・ナレッジFMT更新 ＝ (d)・PoC範囲外）
```

## 3. 機能要件

### 3-1. (a) 自己解決UI（引用付きRAG回答）

- 入力：自由記述の質問文（例：「〇〇タンクは支払い対象でしょうか」）。
- 出力（ナレッジに答えがある場合）：
  - 回答文（F3自社ナレッジの内容のみに基づく。一般知識での補完・推測はしない）。
  - **根拠リスト**：参照した資料名（ファイル／シート／レコード識別子）＋該当箇所の抜粋。
    ユーザーが原本に当たれる粒度で提示する。
- 出力（答えがない場合）：
  - 「ナレッジに該当する情報が見つかりませんでした」の明示＋**起票ボタン**（質問文をプリフィル）。
  - 部分的にしか答えられない場合は、答えられる範囲と不足を区別して提示（不足分は起票導線）。
- 応答時間：即時表示が目標（スライド「3. 画面に役立つナレッジと簡易手順を即時表示」）。数値目標は§9。

### 3-2. (b) 問い合わせ起票・管理（最小限）

- 起票項目：問い合わせNo.（自動採番）・カテゴリ・内容（プリフィル可）・起票者・起票日時。
- ステータス：`受付` → `回答済` → `解決`（最小3状態。優先度・担当割当・通知はPoC範囲外）。
- 電力ユーザー：自分の問い合わせの一覧・ステータス・回答を参照できる。
- NuRO担当者：未回答の問い合わせ一覧を参照し、回答を登録できる。

### 3-3. (c) AI回答ドラフト（NuRO向け）

- 起票された問い合わせに対し、(a) と同一のRAGパイプラインで**回答ドラフト＋根拠**を生成して提示。
- NuRO担当者はドラフトを**参考**に回答を作成する（ドラフトの自動送信はしない。人が必ず介在）。
- (a) で棄却された質問が起票されてくるため、ドラフトも「ナレッジ不在」となるケースが主。
  その場合は検索でヒットした**近傍ナレッジ**（答えではないが関連する資料）を参考情報として提示する。

## 4. RAG設計

### 4-1. 基本方針：3段パイプライン（検索 → 十分性判定 → 引用付き生成＋接地検査）

```
質問
 → ① 検索：Agent Search（F3自社データストア・ハイブリッド検索＋自社フィルタ）
 → ② 十分性判定（sufficiency check）：検索結果でこの質問に答えられるか？を
      回答生成とは独立のLLM判定として実行
      ├─ 不十分 → 棄却応答（「ナレッジに存在しない」）＋起票導線
      └─ 十分 ↓
 → ③ 引用付き回答生成 → 接地検査（grounding check）
      ├─ 接地スコアが閾値未満 → 棄却応答に切替（誤答を出すより棄却）
      └─ 閾値以上 → 回答＋根拠リストを表示
```

**設計根拠（§10 参考文献）**：RAGは文脈を与えることでモデルの見かけの自信を上げ、
文脈が不十分でも棄却せず誤答する傾向が実証されている（Sufficient Context, ICLR 2025）。
同研究は「十分性判定を生成から分離した selective generation」で正答率が改善することを示し、
AbstentionBench (2025) は「モデル任せの棄却は未解決」と結論している。
よって本設計では**②と③後段の二重ゲート**で棄却を構造的に強制する。

### 4-2. 引用（grounding）の実装方式

- 第一候補：**Agent Search の Answer API / Grounded Generation**。
  回答の主張ごとに引用チャンク（ドキュメント＋セグメント）が構造化データで返る。
  既存データストアをそのまま利用でき、引用のでっち上げを構造的に排除できる。
- 接地検査：**Check Grounding API**（回答候補が参照テキストに支持される度合いを0〜1で返す）を
  ③後段のゲートとして利用する方針。閾値は評価（§8）で較正。
- 根拠の表示粒度：F3はExcel由来の構造化レコードのため、
  「ファイル名／シート名／レコード識別子（＋該当フィールドの抜粋）」を最小単位とする。

### 4-3. 検索精度の改善方針（役割分担）

| レイヤ | 手段 | 主担当 |
|---|---|---|
| 再現率（取りこぼし防止） | **投入時の文脈焼き込み**：1レコード＝1ドキュメント生成時に、シート名・分類・案件等の文脈をテキストに前置して索引化（Contextual Retrieval の考え方を前処理で適用。`ingest_knowledge.py` の拡張） | こちらの前処理 |
| 絞り込み | struct_data のメタデータ整備＋**自社フィルタ**（事前レビューの会社名正規化を再利用） | 設定 |
| 適合率・順位 | ブースト・カスタムランキング式・**Ranking API（semantic-ranker）**（事前レビュー §0-3 で採用方針済み） | Google側のノブ |

> **注**：Anthropic の Contextual Retrieval を「そのまま」マネージドの Vertex AI Search 内部に
> 適用することはできない（チャンク分割・埋め込みの間にLLM処理を挟めない）。
> ただし F3 は投入前にこちらでドキュメントを組み立てているため、
> **文脈焼き込みは前処理側で完全に適用可能**。上表の分担が本プロジェクトの実態に即した整理。

### 4-4. 本番拡張への備え

- 本番は「あらゆる社内資料」への横断検索。PoCでは**データストアの追加で資料種別を増やせる**構成を
  維持する（検索対象の指定を設定駆動にし、F3固有のロジックをパイプラインに埋め込まない）。
- 生PDF等の非構造資料を扱う際は、パース済み構造化JSON（テキスト・表・リストのブロック列）の
  持ち込みで前処理側のチャンク制御を確保できる（Agent Search の仕様）。

## 6. データ設計（Firestore・案）

```
inquiries/{inquiry_id}
  ├ number: 自動採番（例 "0001"）
  ├ category: カテゴリ（質問 等）
  ├ content: 問い合わせ内容
  ├ requester: 起票者（電力ユーザー識別子）
  ├ status: "open" | "answered" | "resolved"
  ├ created_at / updated_at
  ├ self_solve_log: 起票前のRAG応答（棄却理由・検索ヒットの記録。分析用）
  ├ ai_draft: { answer, evidences[], grounding_score }   # (c) のドラフト
  └ answer: { content, answered_by, answered_at }         # NuRO回答
```

- `self_solve_log` を残すのは、PoC評価（棄却の妥当性検証・§8）と
  本番の (d)（ナレッジ化）の入力になるため。

## 8. 評価・成功指標（数値は仮置き・要合意 §9）

評価セットは**2群**で作成する（事前レビュー `eval_review.py` のマトリクス評価と同じ流儀）：

- **A群：答えがF3自社ナレッジに存在する質問**（正解の根拠レコードをアノテーション）
- **B群：答えが存在しない質問**（廃炉全般だがF3にない話題・他社F3にしかない話題を含む）

| 指標 | 定義 | 対象群 | 目標（仮） |
|---|---|---|---|
| 正答率 | 正しい内容で回答できた割合 | A群 | ≥80% |
| 引用正確性 | 提示した根拠が実際に回答を支持している割合 | A群 | ≥90% |
| 棄却率 | 「ナレッジに存在しない」と正しく判定した割合 | B群 | ≥90% |
| **誤答率（最重要）** | 答えがないのに回答をでっち上げた割合 | B群 | **≒0%** |

- 誤答率がコア要件（§0-4）の直接検証。棄却率とのトレードオフは
  「疑わしければ棄却（起票に流す）」側に倒す（起票は正規のフォールバックであり失敗ではない）。
- 補助指標：応答時間・②十分性判定と③接地検査それぞれのゲート通過率（閾値較正用）。
