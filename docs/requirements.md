# 要件定義

## 背景・課題

廃炉情報管理システム（NuRO）では、電力会社が委託会社から受け取った工事資料を  
NuRO 提出様式（Excel）に手作業で転記している。

この手作業には以下の課題がある：

- **転記工数が大きい**：1件あたり数十項目を目視で読み取り、手入力する
- **表記揺れによるミス**：委託会社ごとに異なる項目名・日付形式・金額表記に対応が必要
- **手戻りが発生する**：NuRO のレビューで転記ミスが発覚し、修正・再提出が繰り返される
- **属人化している**：担当者の経験に依存しており、引き継ぎコストが高い

---

## PoC ゴール

委託会社から受け取った資料（Excel/Word）をアップロードするだけで、  
NuRO 提出様式への転記が自動完了する仕組みを実証する。

```
【Before】
委託会社資料 → 担当者が目視で読む → 様式に手入力 → NuRO に提出

【After】
委託会社資料 → AI が自動転記 → 担当者が確認・修正 → NuRO に提出
```

電力会社担当者の確認・修正作業は残るが、全量入力の手作業をなくす。

---

## スコープ

| 項目 | スコープ内 | スコープ外 |
|---|---|---|
| 対象様式 | フレーム B（MRC1 シート） | その他シート（MRC2、MOI 等） |
| 対象ファイル形式 | .xlsx、.docx | PDF、画像スキャン |
| 入力ファイル数 | 1ファイルずつ処理 | 複数ファイルの同時処理 |
| UI | CLI スクリプト | Web UI（将来対応） |
| 認証・権限管理 | なし（PoC のため） | ユーザー認証、ロール管理 |

---

## 前任 PoC との差別化

前任 PoC は Knowledge Graph を用いた**学習駆動型**アプローチを採用していた。  
本 PoC は**定義駆動型**アプローチで差別化する。

| 観点 | 前任 PoC（学習駆動） | 本 PoC（定義駆動） |
|---|---|---|
| 精度の源泉 | 過去事例の学習データ | YAML 定義 + synonym 辞書 |
| 初期精度 | 学習データが少ないと低い | YAML があれば即日高精度 |
| 改善方法 | データを蓄積する | YAML の synonym を追記する |
| 保守性 | GraphDB の管理が必要 | YAML ファイルの編集のみ |
| 対象範囲 | 汎用ドキュメント | 廃炉 PJ 業務に特化 |

**核心メッセージ：**  
学習データが貯まるのを待たず、YAML 定義という確実な仕組みで初日から高精度を実現する。

## 将来構想

PoC 検証後は以下の方向に拡張する：

1. **Web UI 化**：ブラウザからファイルをアップロードしてチャット形式で確認・修正
2. **マルチシート対応**：MRC2、MOI 等の他様式への対応（YAML 定義を追加するだけ）
3. **ADK によるマルチ AI エージェント化**：Google Agent Development Kit を用いて  
   data_extractor / cell_locator / review_agent 等を協調動作させる構成に移行
4. **フィードバックループ**：担当者の修正内容を synonym 辞書に自動追記し、精度を継続改善
