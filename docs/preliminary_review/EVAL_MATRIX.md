## 0. 用語集（この文書を読むための最小セット）

| 用語 | この案件での意味 |
|---|---|
| 事前レビュー | 電力会社が提出した様式をNuRO担当者が確認する**前に**、AIが過去ナレッジと照合して指摘の下書きを作る機能（`/review` 画面） |
| RAG | 「検索してから文章を作る」方式。AIの記憶だけで書かせず、**実在するナレッジを検索して根拠にさせる**ことで、もっともらしい嘘を抑える |
| F3ナレッジ | 電力会社とNuROの**過去の審査やり取りの記録**（質問・回答）。費目・炉型・会社名のラベル付き。根拠の主力 |
| F2ナレッジ | **NuRO内部の知見メモ**（問合せ対応・チェックの勘所）。費目ラベルを持たない＝費目横断 |
| 転記結果（mappings） | レビュー対象のデータ。「どのセル番地に何の値が入っているか」の一覧 |
| 様式定義（config） | 様式のどのセルがどの項目かを定義したYAML。チェックの拠り所（特定の費目・セルをプログラムに直書きしない） |
| ハイブリッド検索 | **字面照合**（キーワード一致・BM25）と**意味照合**（文章を数値ベクトル＝埋め込みに変換し近さで照合）を同時に走らせる検索。同義語辞書は持たず、表記ゆれは意味照合が吸収する |
| Reranking／rerankスコア | 検索結果をGoogleの採点AI（semantic-ranker）が「クエリとの関係の深さ」0〜1で採点し直し、点数順に並べ替える工程。**人手の正解ラベルは無く、スコア自体がモデルの判断**（実測で妥当性を校正済み・閾値0.15） |
| 根拠付け（grounding） | AIの指摘に「どの過去事例が根拠か」を紐づけること。**誤grounding**＝無関係な過去事例を根拠として提示してしまうこと |
| ハルシネーション | AIが実在しない事実・記録をもっともらしく生成すること。本件で最も防ぐべき事故 |
| 関連性ガード | AIの出力を**プログラムが機械的に検品する**仕組み（`apply_relevance_guard`）。引用された過去事例が申請の費目と噛み合わなければ、根拠ラベルを外して「AI知見」へ**降格**する（指摘自体は残す） |
| AI知見 | 根拠ナレッジ無しでAIの一般判断として出す指摘のラベル。必ず「要確認」扱いになり、過去事例に基づく指摘（AIからの指摘）と区別される |
| 決定論 | LLMと違い**毎回必ず同じ結果になる**プログラム処理。安全に関わる部分（検品・番地補正・検算）はすべて決定論で作る方針 |
| ハードゲート／provisional | 自動テストの厳しさの区分。**ハードゲート**＝不合格なら開発を止める（exit=1）。**provisional（暫定）**＝正解が未確定のため警告止まり |
| ゴールド指摘 | 「NuROがお手本と認める正解の指摘」。現在は仮説段階で、確定するとテストをハードゲートに昇格できる |
| 過剰適合（作り込み） | テストを通すために特定の資料・ケース専用の処理を作ってしまうこと。本番で通用しなくなるため本件では厳禁 |
| ingest（ナレッジ投入） | ナレッジExcelを検索エンジンに登録する工程（Excel→平坦化→BigQuery→Agent Search索引） |
| Tool1〜5 | 検索・チェックの種別。1=F2／2a=F3自社／2b=F3他社／3=類似工事（範囲外）／4=補足資料（範囲外）／5=計画実績差分（10%閾値のルール） |

## 1. 全体像：レビューが指摘を作るまでの流れ

「**指摘の発見はLLM**（ばらつき許容）」「**出してはいけないものの阻止は決定論**（ばらつきゼロ）」を分離した多層構成。

```
① 検索（ハイブリッド）    クエリ＝申請自身の「費目＋工事件名」。字面照合＋意味照合を同時実行
② Reranking             関係の深い順に並べ替え（件数は削らない）。スコアはF2採用の安全判定にも使用
③ LLM生成（Gemini）      F2/F3を根拠に指摘作成。プロンプトで関連性確認・指摘の統合を指示
④ 決定論の検品（固定順序） 関連性ガード（降格）→ 番地補正 → 出典の可読化
                          → ルール指摘（プレースホルダー・必須欄空欄・Tool5差分）とマージ
```

## 2. 軸① ナレッジ検索精度

### 難1：費目が完全一致するナレッジが上位にヒット
| 観点 | 内容 |
|---|---|
| 担うロジック | ハイブリッド検索の**字面照合側**が費目名を照合し、Rerankingが関連度降順に整列する。評価クエリは費目名そのもの（実運用のクエリは費目＋工事件名） |
| 自動評価 | query「解体撤去費」→ 5件以上ヒット・費目「解体撤去費」を含むこと |
| 実測（2026-07-08） | ✅ PASS 20件。**上位5件すべて費目「解体撤去費」**（score 0.122〜0.164） |
| 達成できない条件 | ①**ナレッジ投入の欠落**＝索引に無いものは絶対にヒットしない。実際に投入時のID衝突で最大49%の行が無音消失（§1-10）、行の捏造（§1-16）が過去に発生。恒久検証は `verify_extraction.py`（生Excelと全件突合） ②ナレッジ側の費目ラベルの表記ゆれ・欠落 ③Reranking API失敗時は検索順のまま素通し＝ヒットはするが上位性が劣化 ④件数増で関連度の打ち切りに遭い圏外へ押し出される（取得上限とセットで調整・§3-1） |
| 乖離・提言 | ①提案資料は「F2,F3」だが評価は**F3のみ**（F2は費目ラベルを持たず「費目完全一致」になじまない→F2用に業務カテゴリ・事象概要ベースの基準を別途定義） ②「**上位に**」の順位は判定条件に入っていない（上位k件以内の基準追加を推奨） |

### 難2：同義語・表記ゆれ（文言が完全一致しなくてもヒット）
| 観点 | 内容 |
|---|---|
| 担うロジック | ハイブリッド検索の**意味照合側**。クエリと文書を埋め込みに変換し、ベクトルの近さで照合する。「解体工事の費用 撤去」と「解体撤去費」は字面が違っても意味が近いためヒットする。**同義語の対応表は持たない**＝メンテ不要だが、吸収できるかはGoogleの埋め込みモデルの語彙知識に依存 |
| 自動評価 | **わざと費目名と一字一句一致しない**クエリ「解体工事の費用 撤去」→ 3件以上・正しい費目群を含むこと |
| 実測 | ✅ PASS 20件。上位5件すべて正しい費目（score 0.18〜0.23） |
| 達成できない条件 | ①モデルが学習していない語彙（廃炉業務特有の略語・社内用語・機器型式など）は吸収されない ②クエリが極端に短い・一般的すぎると意味が定まらず無関係文書が上位を占める ③モデルはGoogleマネージド＝更新で挙動が変わり得る（懸念語彙はYAMLへのケース追加で継続監視） |

### 難3：件数が少ない費目・炉型でもヒット
| 観点 | 内容 |
|---|---|
| 担うロジック | **希少費目**：検索スコアは文書1件ごとにクエリとの関連度を独立採点する方式で、「同種文書の件数が多いほど有利」という項が存在しない＝多数派費目が低スコアで何十件あっても希少費目事例の順位は下がらない。字面照合は出現頻度の低い語の重みを自動で上げるため、希少費目名はむしろ強い判別子になる。**希少炉型**：炉型は検索スコアに頼らず、発電所名→炉型の対応表（`plant_reactor_map.yaml`）から投入時に付与したラベルで、検索後にプログラムが該当炉型以外を機械的に除外する（上限の3倍を先取りしてから絞る） |
| 自動評価 | 希少費目：query「処分費 輸送容器」→ 1件以上・費目「処分費」を含む／希少炉型：PWR指定→**全ヒットがPWR**（1件でも他炉型が混入でNG） |
| 実測 | ✅ PASS（処分費5件・PWR18件で混入ゼロ）。**ただし処分費の本命は4〜5位**（上位3件は近縁の「廃棄物処理費」score 0.20〜0.24、処分費は0.069） |
| 達成できない条件 | ①**意味の近い多数派（近縁費目）が上位を占める**＝母数で沈むのではなく、意味照合が近縁費目にも高スコアを与えることで起きる（実測で観察）。取得上限が小さいと打ち切りで圏外に落ちる ②該当費目の事例がナレッジに0件（検索でなくデータ供給の問題） ③炉型：対応表に発電所が未登録だと炉型ラベルが欠落し、除外フィルタで全件落ちて0件になる ④該当炉型の割合が極端に低いと、3倍先取り（上限100件）に1件も含まれず0件になり得る |
| 提言 | 「正解費目が上位k件以内」基準の追加（本番の件数増では順位こそが品質） |

## 3. 軸② LLMレビュー品質

### 難1：必要な情報のみヒットした際に正しくレビュー生成
| 観点 | 内容 |
|---|---|
| 担うロジック | プロンプトの根拠付け指示：「F3/F2レコードを1件ずつ見て、その過去事例が求めた事項が本様式に記載されているか／十分かを照合し、不足なら参照番号を根拠に指摘する」。検索結果がクリーンならこの照合がそのまま正しい指摘になる |
| 状況 | ✅ 実データの実測で確認済み。ただし**専用の自動評価ケースは意図的に作っていない**（テスト用のクリーン入力を手作りすると過剰適合になり「本番で通用するか」を測る意味が薄れるため） |
| 達成できない条件 | ①**LLMの確率的ばらつき**：検索が当たっていても指摘に変換されない実行回があり得る。初期検証では出力10件すべてが根拠付けゼロになり、プロンプト改修で解消した実例あり（付録A-1）＝プロンプトの書き方ひとつで発火が消える性質 ②**モデル更新による回帰**：Gemini 3.5移行時にレビュー品質の回帰が実際に発生し修正した実績あり。検知網は「F3根拠が1件も出なければ不合格」の自動判定だが、難1単体の自動ケースは無い |
| 対外説明・提言 | **難1単独の自動ケースが無い構造的理由**：本システムは常に複数の検索（F2＋F3自社＋F3他社）を束ねてLLMに渡すため、「必要な情報**のみ**がヒットした」純粋な状態は実運用で発生しない（作るには検索を偽装する必要があり、偽装したテストは実力を証明しない）。したがって難1の能力は**難2（ノイズ混在で正しく生成）の合格に包含**される——ノイズがあっても正しく作れるなら、ノイズが無ければなおさら作れる。対外的には「難1は難2の実証に包含・単独の自動回帰は対象外」と説明する |

### 難2：無関係な情報が混ざっても正しくレビューできる
| 観点 | 内容 |
|---|---|
| 担うロジック | **LLM層**：プロンプトで「根拠に使う前に、その過去事例が本申請の費目・工事と同種か必ず判定せよ」と指示（費目・工事件名を判断材料として明示）。検索は他社・他工事・他話題のレコードも返すため、その中から本申請に関連する事例だけを根拠化できるかを見る |
| 自動評価 | ①**合成ケース（難2専用・2026-07-09追加）**：費目=解体撤去費の小申請。ノイズ混在の検索結果から関連根拠のレビューが出せること（F3根拠1件以上・法令根拠禁止・件数上限） ②実データケース（MRC1/MRC2・難2/3両方の性質を含む） |
| 実測 | ✅ PASS。合成ケース＝指摘5件、F3根拠はタービン建屋関連の実在事例（03_KT_1G_01_0009_01等）を引用。実データ＝MRC1指摘5〜6件・MRC2指摘2〜3件（run毎変動・上限内） |
| 達成できない条件 | ①LLMの確率的ばらつきで根拠化されない run があり得る（性質ベース判定＋暫定WARNで運用） ②「1セル1指摘」のプロンプト指示をLLMが破ることが稀にある（同一セルにF3根拠＋AI知見が並ぶ等。過検出は件数上限で検知） |

### 難3：類似するが無関係な情報（近縁費目）が混ざっても選別できる
| 観点 | 内容 |
|---|---|
| 担うロジック | **決定論層**：関連性ガード＝LLMが引用した事例の費目が申請費目と**語（2文字単位）を共有しなければ**根拠ラベルを外して「AI知見」へ降格（処分費⇔廃棄物処理費は語を共有しない＝不採用／施設解体一解体費⇔解体撤去費は「解体」を共有＝同種）。費目ラベルの無いF2は字面でなく**rerankスコア≥0.15**で判定。LLMが類似ノイズに釣られても出力段階で必ず捕捉される |
| 自動評価 | ①**合成ケース（難3専用・2026-07-09追加）**：費目=処分費の小申請。検索上位を近縁費目「廃棄物処理費」（他社・容器検査の話題）が占める状況（軸①難3で実測済み）で、それを根拠にせず処分費の過去事例のみでレビューできること ②実データケース（MRC1/MRC2） |
| 実測 | ✅ PASS。合成ケース＝指摘6件。**F3根拠の引用は処分費の実在事例（03_KT_1G_01_0011_01＝輸送容器の数量根拠の要求）のみ**で、検索上位3件を占めた廃棄物処理費（03_2G_0008_*）は根拠化されず＝選別の直接証跡 |
| 達成できない条件 | ①**申請の費目が読み取れないと決定論層が働かない**：費目欄が空、または費目欄自体が無いシート（MRC2等）では降格判定をスキップ＝プロンプト指示だけで守る状態になる ②費目の語共有判定の両方向の誤り：**別分野でも語を共有する費目**（「〇〇管理費」同士の「管理」など）は同種と誤判定され降格されない／**語を全く共有しない真の同義費目**は正当な根拠まで降格される。現行の費目体系では未発生だが、費目体系の変更で顕在化し得る ③F2の閾値0.15は現行モデルでの実測校正値＝モデル更新で分布がずれると全降格（警告ログで検知）または過剰採用になる ④Reranking無効・API失敗時はF2判定が字面照合に戻る＝偶然一致の誤判定が復活し得る ⑤LLMが根拠欄に参照番号もレコードIDも書かない場合、引用を辿れず一律降格（安全側だが正当な根拠が消える） |

### 難4：正解不在時に無関係資料・LLM知見で誤ったレビューを生成しない
| 観点 | 内容 |
|---|---|
| 担うロジック | 合成申請（ナレッジに存在しない費目「放射線管理費」）で**検索が実質当たらない状況を意図的に作り**、F3/F2根拠の指摘が1件でも出たら不合格。プロンプト指示に加え、関連性ガードが引用の費目不整合を決定論で降格するため、LLMが指示を破っても出力段階で必ず捕捉される。この決定論保証があるため**マトリクスで唯一のハードゲート**（不合格で開発を止める） |
| 実測 | ✅ PASS。指摘6件すべて「AI知見・要確認」（空欄・抽象表現の客観指摘のみ）＝根拠の捏造ゼロ |
| 達成できない条件（保証の限界） | ①難2/3と同じ前提：**申請の費目が読み取れるシートでのみ**決定論保証が働く ②法令根拠の禁止は「法令・条文・規制・通達」の**字面検知**＝「規則」「基準」等への言い換えはすり抜ける ③**テストケース自体の前提崩れ**：将来「放射線管理費」の実事例がナレッジに追加されると「正解不在」でなくなりケースの組み替えが必要（YAML差し替えのみで対応可能） |

## 4. 実出力の確認方法（信頼性の担保・再現手順）

7項目すべての判定＋**実出力（検索ヒットの中身・指摘の全文）**を1コマンドで表示できる（判定ロジックは不変）：

```bash
uv run python scripts/preliminary_review/eval_review.py --verbose                 # 全9項目＋根拠表示
uv run python scripts/preliminary_review/eval_review.py --verbose --case 難2_合成 # 1項目だけ証跡出力（ケース名の部分一致）
uv run python scripts/preliminary_review/eval_review.py                      # 判定のみ
PYTHONPATH=. uv run pytest apps/backend/tests/test_review_e2e.py             # 回帰テスト
```

- `--case` は項目単位で証跡を取る用途（例：`--case MRC2`／`--case 難4`）。一致なしの場合は定義済みケース一覧を表示して終了する。ファイルに残す場合は `> data/verification/証跡_難1.txt` 等へリダイレクト（`data/verification/` はコミット対象外）。

証跡は**入力→合格条件→出力**が1画面で揃う（出力だけでは検証にならないため）：
- 検索4項目：**入力**（検索クエリ・炉型/会社フィルタ・取得上限）と**合格条件**（min_hits・fee_any 等）に続けて、ヒットごとに**順位・rerankスコア・費目・炉型・会社・メッセージID・内容**。
- レビュー3項目：**入力**（レビュー対象Excelパスまたは合成データの全セル値・検索文脈＝会社/費目/炉型・転記結果の件数と抜粋）と**合格条件**に続けて、指摘ごとに**重要度・セル番地・根拠種別（F3/AI知見）・指摘文・出典**。難4は合成5セルが全件表示されるため「F3に存在しない費目を入力→F3根拠ゼロ」の因果がそのまま読める。

## 5. 本番開発への引き継ぎ（3分類）

PoCの成果物は「マトリクス全PASSのシステム＋再現ハーネス＋本書」。以下は**放置した場合の影響で3分類**した引き継ぎ事項。
出典：各項目は `RAG_VERIFICATION.md §2`（バックログ正本）またはコードの該当箇所で確認済み。

### ① 本番開発で実装が必須（やらないと実害が出る）

| 項目 | かみ砕くと | 放置するとどうなるか | 出典 |
|---|---|---|---|
| 添付資料との突合（Tool4） | 電力会社が様式に添付する見積書・内訳明細書の中身と、レビュー結果を突き合わせる機能。PoCでは範囲外 | 積算根拠が添付に実在しても様式に無ければ「根拠が確認できない」と指摘する＝**誤指摘が常態化**し担当者の棄却の手間が増える | §2④ |
| 費目が無いシートでの検品有効化 | 「無関係な過去事例を根拠にしていないか」の機械検品（関連性ガード）は、**申請の費目が読み取れるシートでしか働かない**。MRC2のように費目欄が無いシートでは素通し。基本情報シートから費目を引き継げば有効化できる | 費目なしシートだけ、誤った根拠付けをプロンプト指示（AIの自制）だけで抑えている状態が残る | コード確認（`review_logic.py` 降格判定） |

| 計画実績差分の対象定義（Tool5） | 計画と実績の乖離（10%超）を検出する仕組み自体は完成しているが、**比較するセルのペアが様式定義に未登録**のため実質休眠 | 実績報告時の乖離チェックが機能しない（定義を書けばコード変更なしで自動復活） | §2①b |
